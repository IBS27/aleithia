1. Where the project stands today

Aleithia is a data aggregation and analytics platform. Sixteen pipelines (news, Legistar, Federal Register, Reddit, Yelp, Socrata permits/licenses/inspections, CCTV, vision, parking, and so on) all normalize into one shape, the Document model in modal_app/common.py:

class Document(BaseModel):
    id: str
    source: SourceType        # news, politics, yelp, public_data, ...
    title: str
    content: str
    metadata: dict            # flexible, source-specific
    geo: dict                 # neighborhood, ward, lat/lng
    status: str               # "raw" → "enriched" → "graphed"

On top of those documents, modal_app/graph.py builds a "city knowledge graph": 4 node types (neighborhood, regulation, entity, business_type) and 3 edge types (regulates, sentiment, competes_in). It is served by /graph/full and /graph/neighborhood/* and rendered in the frontend.

This graph looks like a knowledge graph but does not behave like one. Look at how a node gets its identity:

# graph.py — a "regulation" is just a slugified title
reg_id = f"reg:{title[:50].replace(' ', '-').lower()}"

# an "entity" is just a slugified business name
ent_id = f"ent:{entity_name[:40].replace(' ', '-').lower()}"

The node ID is a string mangling of display text. Two news articles about "Lou Mitchell's" and a Yelp record for "Lou Mitchells Restaurant" produce different nodes. A regulation whose title is truncated differently on re-ingest produces a duplicate node. There is no notion of "the same thing" anywhere in the system. Edge weights are heuristics (age decay, sentiment magnitude, count caps) chosen for visual layout, not for meaning — city-wide regulations are even linked to ten hard-coded "representative neighborhoods" so the picture doesn't look empty.

None of this is a criticism of the code. It does its job, which is to produce a visualization. The point is that the graph is an output for the eye, not a substrate for queries or reasoning. Everything that involves actual meaning — risk categories, sentiment, synthesis — is done by ML models (BART, RoBERTa) and LLM agents over raw document text, with Supermemory doing embedding search on the side.

2. The core problem: documents, not entities

One sentence captures the whole gap:

Aleithia models documents about things. An ontology models the things, and treats documents as evidence about them.

Every shortcoming traces back to this. Consider one real restaurant in Pilsen. Today it exists in the system as scattered, unconnected documents:

SourceWhat exists todayKnows about the others?
Business license (Socrata)A public_data document with legal_name, license_description in metadata.raw_recordNo
Food inspection (Socrata)A separate document with dba_name, risk, violationsNo
Yelp reviewsDocuments with business_name, rating in metadataNo
News mentionA document whose content happens to contain the nameNo
Building permitA document with an address, no name linkNo

A human reading all five would immediately say "that's one business, and it's in trouble: failed inspection, declining reviews, and a renovation permit suggesting an ownership change." The system cannot say that, because nothing in it represents the business. The LLM agents can sometimes stitch this together from retrieved text, but the result is non-deterministic, unverifiable, and recomputed from scratch every time.

An ontology is the fix for exactly this: a layer of entities with stable identity, explicit types, and typed relationships, where each fact points back to the documents that support it.

3. The five changes that would make it an ontology

3.1 An entity layer with real identity

This is the foundation; nothing else matters without it. Introduce entity records that live alongside documents — not derived on the fly inside build_city_graph(), but persisted, with stable IDs:

class Business(BaseModel):
    id: str                        # stable, e.g. "biz:lou-mitchells-0042"
    canonical_name: str
    aliases: list[str]             # "Lou Mitchell's", "LOU MITCHELLS INC"
    license_numbers: list[str]
    address: str | None
    neighborhood: str              # FK to Neighborhood entity
    naics_code: str | None
    source_document_ids: list[str] # provenance: every doc that mentions it

Creating these requires an entity resolution step in the enrichment pipeline: when a new document arrives, decide whether the business it mentions is one we already know. The Chicago data makes this tractable — license records carry license numbers and addresses, inspections carry dba_name + address, permits carry addresses. Address + fuzzy name matching resolves most of the public-data records deterministically; an LLM call can adjudicate the ambiguous tail (it's a good use of a model: a narrow yes/no with evidence, not open-ended synthesis).

The same treatment applies to regulations: a Legistar ordinance should be one entity keyed by its record number, no matter how many documents mention it — not a slug of its truncated title.

3.2 An explicit schema: classes and typed relations

Today the four node types and three edge types exist only as string literals inside graph.py. An ontology makes the schema a first-class, documented artifact: which entity classes exist, which relations are allowed between which classes, and what each relation means.

A realistic starting schema for this domain — small on purpose:

ClassIdentity comes from
NeighborhoodThe fixed list of 77 community areas (already exists)
BusinessEntity resolution over licenses, inspections, reviews, news
BusinessTypeControlled vocabulary (see 3.3)
RegulationLegistar record number / Federal Register document number
Permit / Inspection / LicenseTheir Socrata record IDs (these are events, not just documents)
RelationDomain → RangeReplaces
located_inBusiness → Neighborhoodthe generic sentiment edge
has_typeBusiness → BusinessType
licensed_asBusiness → Licensenothing — these don't exist today
inspected_inInspection → Business
permitted_atPermit → Business / address
applies_toRegulation → BusinessType | Neighborhoodthe regulates edge with its fake city-wide links

Note what disappears: the current regulates edge guesses geography from text mentions and pads with hard-coded neighborhoods. applies_to would assert what a regulation actually governs — which is a harder extraction problem, but it's the difference between a picture and a claim.

3.3 Controlled vocabularies instead of free strings

Today business_type nodes come from raw license_description strings, title-cased and truncated to 30 characters. "Retail Food Establishment" and "Retail Food Establishm" can coexist; categories from Yelp never align with categories from licenses. Risk categories ("zoning", "competition", "sentiment" in RiskFactor.category) are likewise free strings.

The fix is small but high-leverage: a fixed mapping table from raw license descriptions and Yelp categories onto one canonical taxonomy (NAICS codes are the obvious choice — Chicago license data is finite, so the mapping is a few hundred rows you write once), and an enum for risk categories. This is the cheapest item on the list and it immediately makes cross-source comparison meaningful: "restaurants" from licenses, inspections, and Yelp finally refer to the same set.

3.4 Provenance: every fact traces to its evidence

The skeleton already exists — RiskFactor has evidence_ids, and Document.status tracks a lifecycle. The ontology version generalizes it: every entity attribute and every relation carries the document IDs (and extraction method) it was derived from.

{ "subject": "biz:lou-mitchells-0042",
  "relation": "inspected_in",
  "object": "insp:2026-04-12-1187",
  "evidence": ["doc:public_data:abc123"],
  "extracted_by": "socrata-direct",      # vs "llm" — trust differs
  "asserted_at": "2026-04-13T..." }

This is what turns "the model says risk is 7.2" into "risk is 7.2 because of these three inspections and this pending ordinance — click to see them." For a product whose pitch is democratizing business intelligence for small operators, inspectable claims are the difference between a toy and a tool.

3.5 A queryable store, with rules where rules beat models

Once entities and typed relations exist, the graph stops being a JSON file regenerated for display and becomes something you query. This does not require a triple store or SPARQL — SQLite/Postgres tables for entities and relations, or even structured JSON on the existing Modal volume with indexed lookups, is enough at this data size.

It also lets some intelligence move from LLM inference to deterministic rules where determinism is better:

# Today: an LLM reads retrieved text and may or may not notice this.
# With an ontology, it's a query that always returns the same answer:

failed_inspections(b) AND b.has_type = "restaurant"
  AND pending_regulation applies_to b.neighborhood
  → risk_factor(b, category="regulatory", severity="high",
                evidence=[...])

The LLM agents don't go away — they get better inputs. Instead of feeding an agent a pile of retrieved document text, you feed it the resolved entity neighborhood: "here is this business, its inspection history, its license status, the three regulations that apply to its type, and the recent sentiment trend — synthesize." Less hallucination room, smaller prompts, reproducible grounding.

4. How the product changes

The clearest way to see the difference is in what a user can ask.

"Show me food businesses in Pilsen with failed inspections that are affected by pending ordinances."
Today: impossible as a query. An agent might approximate it from retrieval, differently each run. After: a deterministic join across has_type, located_in, inspected_in, applies_to, with every row linking to its source documents.
"Why is the risk score for coffee shops in Logan Square 6.8?"
Today: RiskFactor descriptions written by a classifier, evidence lists best-effort. After: each factor decomposes into named entities and relations — this ordinance, these four recent closures, this permit cluster — each clickable to its evidence.
"What's happening with this specific business?"
Today: there is no page for a business; there are search results that mention a name. After: an entity page is almost free — license status, inspection history, review trend, news mentions, nearby permits, all attached to one stable ID. This is a new product surface that the current architecture structurally cannot offer.
The graph view itself.
Today: decorative — duplicate nodes, padded edges, weights tuned for layout. After: clicking an edge shows a claim and its evidence. The visualization becomes a window onto the data instead of an artifact generated from it.
Agent output quality (synthesis, impact briefs, lead analyst).
Today: agents reason over retrieved text; quality depends on what retrieval surfaced. After: agents receive resolved, structured context. Impact briefs cite entities and relations rather than paraphrasing documents, so two runs over the same data agree with each other.

And one honest non-change: dashboards that are pure document feeds — news lists, Reddit, TikTok trends, CCTV counts — look the same. The ontology pays off where sources intersect, not where one source is displayed.

5. The actual work, in order

Sequenced so each phase is independently useful and nothing depends on a later phase:

PhaseWorkTouchesRough size
1. Vocabularies NAICS mapping for license descriptions and Yelp categories; enum for risk categories. Pure data + a lookup function. modal_app/common.py, enrichment pipeline Days
2. Schema + entity store Pydantic classes for Business, Regulation, Permit, Inspection, License; relation records with provenance; storage on the existing volume or in Postgres. New modal_app/ontology/ (or similar), backend/shared_data.py accessors ~1 week
3. Entity resolution Deterministic pass first (license numbers, Socrata IDs, address + name fuzzy match), LLM adjudication for the ambiguous tail. Runs as a new enrichment stage; Document.status gains a "resolved" step. Enrichment pipeline, new resolution module 2–3 weeks; this is the hard part, and match-quality evaluation is most of it
4. Rebuild the graph on entities build_city_graph() reads the entity store instead of re-scanning documents. Same JSON contract to the frontend (per repo rules, keys preserved), but node IDs become stable entity IDs and edges become typed relations. modal_app/graph.py, graph routes ~1 week
5. Query layer + rules Internal query functions over entities/relations; port the most valuable risk heuristics to deterministic rules; feed structured entity context to agents. modal_app/agents.py, risk scoring, new query module 1–2 weeks
6. Product surface Business entity pages, evidence links on risk factors, claim popovers on graph edges. Frontend, new API routes Ongoing

Total: roughly 6–8 focused weeks to reach "the graph is real and queries work," with phases 1–2 delivering visible cleanup almost immediately. The ongoing cost is real too: entity resolution needs monitoring (bad merges are worse than no merges), and the schema needs an owner — every new pipeline must now answer "which entities and relations does this produce?" instead of just "which documents?"

6. What to avoid

  • OWL, RDF, triple stores, SPARQL. The value here is identity + typed relations + provenance, not description-logic inference. Semantic-web tooling would add operational weight (a new store to run inside Modal's volume-based architecture) for machinery this domain doesn't need. Plain tables and Pydantic get all the benefit.
  • Ontologizing everything. TikTok audio, CCTV frames, traffic counts are signals, not entities. Leave them as documents/timeseries. Five entity classes that are correct beat twenty that are half-maintained.
  • LLM-first entity resolution. The Socrata data has license numbers and addresses; use them. Models adjudicate the residue, deterministic keys do the bulk. The reverse ordering is slow, expensive, and silently wrong.
  • A big-bang migration. The document pipeline stays exactly as it is; the entity layer is built from documents as an additional enrichment stage. If resolution breaks, the current product keeps working.

7. Is it worth it?

Depends on what Aleithia is supposed to become.

If the product is a monitoring dashboard — feeds, trends, sensor analytics, an LLM summary on top — the current architecture is honestly fine, and an ontology is a quarter of engineering spent on internals the user never sees. Skip it; maybe do phase 1 (vocabularies) anyway because it's days of work and fixes visible inconsistencies.

If the product is what the README claims — data-driven business intelligence a small operator can act on — then the ontology isn't an optional upgrade, it's the missing core. Intelligence a user will bet a lease on must be specific ("this business, this ordinance, these inspections"), reproducible (the same question gets the same answer tomorrow), and inspectable (every claim traces to a public record). Documents plus embeddings plus LLM synthesis can't deliver those three properties. Entities, typed relations, and provenance are precisely the machinery that does — and the cross-source data Aleithia already ingests is unusually well-suited to it, because Chicago's public records carry the join keys.

One-line version: Aleithia today knows what was said. An ontology would make it know what is — and for this product, that's the difference that matters.
