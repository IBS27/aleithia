Aleithia architecture brief

Modal workload map and migration plan

Aleithia already has a backend in backend/. The confusing part is that Modal also hosts a second FastAPI app in modal_app/web.py. Migration should consolidate portable Modal routes into the existing backend, then replace Modal storage, queues, GPU jobs, and sandbox primitives in stages.

Current State

Recommendation: keep backend/ as the main API, move read-heavy Modal routes into that existing backend first, keep GPU jobs on Modal until they are benchmarked against alternatives, and move the shared dataset from Modal Volume to object storage before replacing worker primitives.
API clarification: this document uses "API migration" to mean removing the second Modal-hosted FastAPI surface over time. It does not mean creating a new backend or abandoning backend/main.py.

Two API Surfaces

backend/main.py is the local backend and owns /api/data/*. modal_app/web.py builds another FastAPI app and exposes richer routes with @modal.asgi_app(). The migration target is not a new API; it is route consolidation into backend/.

backend/main.py modal_app/web.py /neighborhood /analyze /vision/* /parking/* /graph/*

Runtime Data

The canonical runtime dataset is the Modal Volume alethia-data, mounted as /data. Code writes raw documents under /data/raw and derived outputs under /data/processed.

Target Architecture

The target should keep the existing backend as the API consolidation point and separate normal dashboard requests from bursty compute. Hosting choices such as Cloud Run, ECS, Fly, Render, or App Runner only matter when deploying that existing backend; they are not replacements for the backend code already in the repo.

Recommended Hybrid

  • API: existing backend/ app with portable Modal routes folded into it.
  • Backend hosting: keep local for development; deploy the same backend to Cloud Run, ECS, Fly, Render, or App Runner if production hosting is needed.
  • Storage: S3, GCS, or R2 for raw JSON, processed JSON, and images.
  • Queues: SQS, Pub/Sub, Redis Streams, or Cloud Tasks.
  • State: Redis, DynamoDB, Postgres, or Firestore for cost and refresh dictionaries.
  • GPU: keep Modal first, then benchmark RunPod, Cloud Run GPU, and AWS Batch.
  • Sandbox: E2B or isolated container jobs for generated code execution.

Why Not Move Everything At Once

  • Modal Volume paths are embedded in most worker modules.
  • Queue behavior is coupled to classifier and Lead Analyst flow.
  • GPU classes rely on Modal snapshots, batching, and per-call dispatch.
  • modal.Sandbox is a security boundary, not just a subprocess wrapper.
  • Moving all workloads together makes cost and latency regressions hard to isolate.

Migration Plan

Treat this as staged consolidation, not a platform swap. The existing backend/ becomes the main API surface, while workers, GPU jobs, and sandbox execution remain separate services behind stable contracts.

  1. Set the baseline. Measure endpoint latency, Modal function invocations, GPU seconds, queue volume, storage size, OpenAI calls, failure rate, and source freshness. Classify each Modal path as a read route, CPU job, queue worker, GPU job, sandbox job, or agent flow.
  2. Make storage portable first. Move the raw documents, processed documents, generated images, impact briefs, and file-backed state toward object storage. Keep reads and writes behind a storage abstraction so routes and workers do not care whether the data is coming from Modal Volume, S3, GCS, or R2.
  3. Consolidate read routes into backend/. Move Modal routes that only read JSON or images into the existing backend. Start with graph, parking reads, CCTV reads, streetscape summaries, social summaries where fallback behavior is clear, and impact-brief list/detail routes. Keep the frontend response contracts unchanged.
  4. Move CPU ingestion jobs. Convert CPU-only ingesters into standalone scheduled jobs. Each job should fetch external data, normalize documents, write to object storage, enqueue classification work, update freshness metadata, and retry safely.
  5. Replace queues and small shared state. Put queue and dictionary operations behind adapters. Use SQS, Pub/Sub, Redis Streams, Celery, or Cloud Tasks for queues, and Redis, DynamoDB, Firestore, or Postgres for refresh locks, restart backoff, and cost/freshness metadata.
  6. Keep GPU on Modal until benchmarked. Do not move GPU first. Benchmark document classification, sentiment, CCTV YOLO, parking analysis, and Whisper transcription independently. Move a GPU workload only if the new platform is clearly cheaper, faster, or easier to operate for that specific job.
  7. Move sandbox and agent flows last. Keep generated-code execution isolated with E2B or another real sandbox boundary. Move agent fan-out only after storage and queues are stable, and preserve the contracts for /analyze, /command/synthesis, /brief/*, and impact analysis.

Cutover Method

  1. Run the old Modal path and the new path side by side.
  2. Compare response shape, status codes, data counts, freshness, latency, and error rate.
  3. Switch traffic with a config flag or frontend base URL change, starting in local or internal use.
  4. Keep the Modal fallback until the new path is stable, then remove the old route or worker.

Important Non-Goal

Do not collapse everything into the web backend. The backend should serve API requests. Workers should run separately. GPU jobs should stay isolated. Sandbox execution should stay isolated. Putting all of that into one server would make the app harder to operate and slower under load.

Risks and Controls

Migration Risks

  • Endpoint contracts drift while moving route ownership.
  • Object storage listing semantics differ from filesystem traversal.
  • Queue retries can duplicate documents without idempotent writes.
  • Generated code execution can become unsafe if sandbox isolation weakens.
  • GPU migration may increase latency if model load and artifact transfer are slower.

Controls

  • Snapshot response contracts before moving each endpoint.
  • Use adapter interfaces for storage, queues, and dictionary state.
  • Keep document IDs deterministic and writes idempotent.
  • Run old and new endpoints side by side during cutover.
  • Benchmark per workload instead of choosing one GPU provider globally.

Decision Gates

Gate Pass criteria
Storage move Source counts, latest timestamps, image retrieval, and key dashboards match Modal-backed results.
Route consolidation P95 latency improves or stays flat, error rate is unchanged, and frontend views require no contract changes after Modal-owned read routes move into backend/.
CPU jobs move Jobs write the same document schema, retry safely, and update freshness within current thresholds.
GPU move Total cost per successful batch is lower or latency is materially better without lowering success rate.
Sandbox move Generated code remains isolated, network access is controlled, output schema is preserved, and timeout behavior is predictable.

Step 2 Storage Implementation

The backend shared-data layer now has an explicit S3-compatible object-storage accessor beside the existing Modal Volume accessor, and the Modal document service resolves raw and processed JSON reads through that same shared-data layer. Modal remains the default. Object storage is only used when configured through environment variables, so local and Modal-backed reads do not silently switch data roots.

Implemented: backend/shared_data.py can list, stat, read, and write shared paths through either modal:// or s3:// roots while preserving the existing raw and processed path contract. modal_app/api/services/documents.py now uses those shared roots for document, demographics, CTA cache, and source freshness reads.
Setting Purpose Example
ALEITHIA_SHARED_DATA_BACKEND Selects the storage backend. Defaults to modal. s3
ALEITHIA_OBJECT_STORAGE_BUCKET Bucket that contains the runtime dataset. alethia-data
ALEITHIA_OBJECT_STORAGE_PREFIX Optional prefix before raw/ and processed/. runtime
ALEITHIA_OBJECT_STORAGE_ENDPOINT_URL Optional S3-compatible endpoint for R2, MinIO, or non-default S3 endpoints. https://account.r2.cloudflarestorage.com
ALEITHIA_OBJECT_STORAGE_REGION Signing region. Defaults to us-east-1. auto for some R2 setups, or an AWS region for S3.
ALEITHIA_OBJECT_STORAGE_ACCESS_KEY_ID and ALEITHIA_OBJECT_STORAGE_SECRET_ACCESS_KEY S3-compatible credentials. Standard AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY are also accepted. Store as deployment secrets, not in source control.

This completes the first storage portability cut for backend shared reads. The next storage work is to move Modal worker writes off direct Path("/data/...") operations in small groups, starting with low-risk generated JSON outputs before image-heavy or GPU-owned artifacts.
